---
title: "Agent boot documents"
description: "Give agents shared context they read at the start of every session."
---

Boot documents are regular HQ documents that get injected into an agent's context automatically when it wakes up. Use them to give agents durable context they should always have — without repeating yourself every conversation.

---

## What boot documents are for

Agents read their boot documents before doing any work. This means you can:

- Tell an agent about your company without explaining it every time
- Define a writing style or tone it should always use
- Share a list of current priorities or active projects
- Set standing rules ("always CC me on outbound emails", "never commit to pricing without asking")
- Describe standard procedures the agent should follow for recurring tasks

Think of boot documents as the agent's permanent briefing — the things it should always know, regardless of what conversation it's in.

---

## How to tag a document

<Steps>
  <Step title="Create or open a document">
    Go to **Documents** in the left sidebar and create a new document, or open an existing one.
  </Step>
  <Step title="Add a boot tag">
    Tag the document with one of the boot prefixes:

    | Tag | Scope |
    |---|---|
    | `boot:all` | Injected for **every agent** in your workspace |
    | `boot:<agent-slug>` | Injected for **one specific agent** only |

    For example, a company one-pager tagged `boot:all` will be read by every agent. A detailed brief about your newsletter cadence tagged `boot:newsletter-writer` will only be read by that agent.
  </Step>
  <Step title="Agent reads it automatically">
    The next time the agent wakes up (via a message, a task, or an automation), it reads the tagged document as part of its startup context. No restart required — changes take effect on the next session.
  </Step>
</Steps>

---

## Keeping boot documents effective

<Warning>
  Large boot documents slow agents down and reduce their effectiveness. Context has a limit — the more you pack in, the less room the agent has to think about the actual task.
</Warning>

**Keep each document focused.** One document per topic is better than one giant document with everything. An agent reads all tagged documents, so you can have multiple short ones.

**Keep them current.** An agent reading a stale document ("Q1 priorities" in Q3) is worse than reading nothing — it acts on outdated information. Review boot documents quarterly or when your situation changes significantly.

**Signs a boot document is stale:**
- The agent references old project names or priorities
- It uses outdated context in responses
- It contradicts what you've told it recently in conversation

To update: edit the document directly — changes take effect on the next session.

## Good boot document examples

<AccordionGroup>
  <Accordion title="Company one-pager (boot:all)">
    Who you are, what the company does, who the customers are, your stage, your voice. 3–5 paragraphs maximum. Every agent benefits from this context.
  </Accordion>
  <Accordion title="Writing style guide (boot:all or boot:specific-writer)">
    Tone, vocabulary, formatting preferences, what to avoid. Examples of good and bad outputs help more than abstract rules.
  </Accordion>
  <Accordion title="Current quarter priorities (boot:all)">
    What matters right now — 3–5 bullet points. Update at the start of each quarter. Helps agents prioritize when they have discretion.
  </Accordion>
  <Accordion title="Standing rules (boot:all)">
    Hard rules that should never be broken: "never commit to pricing", "always CC [name] on outbound", "don't share confidential deal terms". Short list, no ambiguity.
  </Accordion>
  <Accordion title="Standard procedures (boot:specific-agent)">
    Step-by-step workflows for recurring tasks — e.g., how to research a new contact, how to structure a newsletter draft, how to handle inbound inquiries. Role-specific, so tag per agent.
  </Accordion>
</AccordionGroup>
